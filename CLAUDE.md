# CLAUDE.md

이 파일은 이 프로젝트 전체에서 항상 지켜야 할 공통 규칙을 담는다. 뉴스 기능만의 세부 규칙은 `news/CLAUDE.md`에 별도로 정의한다.

## 🚨 최우선 보안 규칙

**네이버 API 키(Client ID/Secret)는 어떤 경우에도 프론트엔드 코드에 절대 넣지 않는다.**

- 프론트엔드(React) 코드, 환경 변수 중 브라우저에 번들되는 값(예: `VITE_`, `NEXT_PUBLIC_` 등 클라이언트 노출 접두사가 붙은 변수), 커밋되는 소스 파일, 클라이언트 로그 어디에도 API 키를 작성하지 않는다.
- API 키는 반드시 Supabase Edge Function의 서버 사이드 환경 변수(Secrets)로만 관리한다.
- 프론트엔드는 `supabase.functions.invoke(...)` 등을 통해 Edge Function이 가공한 결과만 전달받으며, 원본 API 키나 인증 헤더를 직접 다루지 않는다.
- 코드를 작성하거나 리뷰할 때 API 키가 실수로라도 클라이언트 번들에 포함되지 않는지 항상 확인한다.

## 실패 허용 원칙

- 부가 기능(뉴스 브리핑 등 알람의 핵심 동작이 아닌 기능)이 실패하거나 오류가 나더라도, 알람 소리 자체는 반드시 정상적으로 울려야 한다.
- 부가 기능의 실행 경로는 알람의 핵심 동작(정해진 시각에 소리 재생)과 분리하여 구현하고, 부가 기능에서 발생한 예외가 핵심 동작을 막거나 지연시키지 않도록 한다.

## 데이터 처리 원칙

- 뉴스 데이터는 DB에 저장하지 않는다. 매 요청마다 실시간으로 조회한 결과만 사용하고, 서버/클라이언트 어디에도 영속 저장하지 않는다.
- Edge Function 로그 등 어디에도 API 키, 인증 헤더 등 민감 정보를 기록하지 않는다.

## 아키텍처 원칙

- 외부 API(네이버 뉴스 검색 API 등)를 호출할 때는 브라우저에서 직접 호출하지 않고, Supabase Edge Function을 프록시로 사용하여 CORS 문제와 API 키 노출 문제를 함께 해결한다.
- Edge Function의 CORS 설정은 필요한 출처로 제한하는 것을 원칙으로 하며, 운영 환경에서 `Access-Control-Allow-Origin: *`를 그대로 사용하지 않는다.
- 외부 API 호출에는 타임아웃을 설정하여, 응답 지연이 화면 표시나 알람 동작을 블로킹하지 않도록 한다.

## 작업 순서 원칙

새로운 기능을 구현할 때는 다음 순서를 따른다.

1. 먼저 화면(UI)을 가짜 데이터(mock data)로 완성한다.
2. 그다음 실제 백엔드(Supabase Edge Function 등)를 연결한다.
3. 예외 처리(에러 상태 표시, 실패 허용 로직 등)는 가장 마지막에 추가한다.

## 미확정 사항 처리 원칙

- `news/prd.md`의 Open Questions(예: 뉴스 검색 키워드 방식, 음성 브리핑 재생 타이밍)는 아직 확정되지 않았다. 관련 코드를 작성할 때 임의로 결정하지 말고, 확정이 필요하면 반드시 사용자에게 먼저 확인한다.
