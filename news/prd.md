# PRD: 알람 화면 최신 뉴스 브리핑 기능

> 기반 문서: `최근뉴스_기술명세서.md` (기존 알람 웹앱에 "최신 뉴스 5개 검색 및 음성 읽어주기" 기능을 Supabase + React 스택으로 추가)

## 1. 배경 및 목표

기존 알람 웹앱은 설정한 시각에 소리로만 사용자를 깨우는 단순한 기능만 제공한다. 사용자가 알람을 끄고 하루를 시작하는 시점에 주요 뉴스를 자연스럽게 접할 수 있다면, 별도로 뉴스 앱을 열어보는 수고 없이 아침 루틴 안에서 정보 습득이 가능해진다.

이번 기능의 목표는 알람 화면에 최신 뉴스 5건을 텍스트로 보여주고, 이를 음성(TTS)으로 읽어주는 것이다. 뉴스 조회는 네이버 뉴스 검색 API를 Supabase Edge Function을 통해 프록시하여, API 키 노출과 CORS 문제를 동시에 해결하는 구조를 취한다.

이 기능은 어디까지나 "알람의 부가 기능"이며, 알람 본연의 기능(정해진 시각에 소리로 깨우기)을 절대 저해해서는 안 된다는 것이 가장 중요한 전제다.

## 2. 사용자 스토리

- 사용자로서, 알람이 울렸을 때 화면에서 최신 뉴스 5개 제목을 바로 확인하고 싶다. 그래야 잠에서 깨며 오늘의 주요 소식을 빠르게 파악할 수 있다.
- 사용자로서, 뉴스 내용을 눈으로 읽지 않아도 음성으로 들을 수 있길 원한다. 그래야 씻거나 옷을 갈아입는 동안에도 뉴스를 들을 수 있다.
- 사용자로서, 인터넷 연결이 불안정하거나 뉴스 API가 응답하지 않는 상황에서도 알람 소리만큼은 반드시 울리길 원한다. 그래야 뉴스 기능 오류 때문에 늦잠을 자는 일이 없다.
- 사용자로서, 내 네이버 API 인증 정보가 외부에 노출되지 않길 원한다(직접적으로 인지하지 않더라도, 서비스 운영자로서 이 요구는 반드시 지켜져야 한다).

## 3. 기능 요구사항

| ID | 요구사항 |
|---|---|
| FR-1 | 알람이 울리는 시점(또는 알람 화면 진입 시점)에 최신 뉴스 5건을 조회한다. |
| FR-2 | 뉴스 조회는 Supabase Edge Function(`get-news`)을 통해 네이버 뉴스 검색 API를 호출하는 방식으로 구현한다. |
| FR-3 | 조회된 뉴스 5건의 제목(및 필요 시 요약)을 알람 화면에 리스트로 표시한다. |
| FR-4 | Web Speech API(`SpeechSynthesisUtterance`, `lang: 'ko-KR'`)를 사용해 뉴스 제목/본문을 음성으로 읽어준다. |
| FR-5 | 뉴스 조회가 실패하거나 지연되는 경우, 화면에는 에러/대체 상태를 표시하되 알람 소리 재생 로직에는 영향을 주지 않는다. |
| FR-6 | 뉴스 목록과 TTS 재생은 매번 알람 화면 진입 시 새로 조회한 결과를 기준으로 동작한다(이전 결과를 재사용하지 않음). |

## 4. 비기능 요구사항 (보안 최우선)

**보안**
- 네이버 API 키(Client ID/Secret)는 프론트엔드 코드, 번들, 네트워크 응답, 브라우저 개발자 도구 등 그 어디에도 노출되어서는 안 된다. 반드시 Supabase Edge Function의 서버 사이드 환경 변수(Secrets)로만 관리하고, 프론트엔드는 `supabase.functions.invoke('get-news')` 호출 결과(가공된 뉴스 JSON)만 전달받는다.
- Edge Function 응답에는 API 키, 인증 헤더 등 민감 정보가 절대 포함되지 않도록 응답 페이로드를 검증한다.
- Edge Function의 CORS 설정은 필요한 출처(origin)로 제한하는 것을 원칙으로 하며, 운영 환경에서 `Access-Control-Allow-Origin: *`를 그대로 사용하지 않는다.

**데이터 처리**
- 뉴스 데이터는 DB(Postgres, `news_cache` 등)에 저장하지 않는다. 매 요청마다 네이버 뉴스 API를 실시간으로 호출하여 결과를 반환하고, 응답 후에는 서버/클라이언트 어디에도 영속 저장하지 않는다.

**가용성/안정성**
- 뉴스 브리핑 기능(조회, 표시, TTS 재생) 중 어떤 단계에서 실패하더라도, 알람 소리 자체는 반드시 정상적으로 울려야 한다. 알람 오디오 재생 로직은 뉴스 기능과 완전히 분리되어(독립된 실행 경로) 뉴스 관련 예외가 알람 소리 재생을 막거나 지연시키지 않도록 구현한다.
- 뉴스 API 호출에는 타임아웃을 설정하여, 응답 지연이 알람 화면 표시나 알람 동작을 블로킹하지 않도록 한다.

**로깅/모니터링**
- Edge Function 로그에 API 키, 인증 헤더 등 민감 정보를 기록하지 않는다.

## 5. 범위 제외 (Out of Scope)

- Supabase Auth를 통한 로그인/회원가입 기능
- 사용자별 알람/뉴스 설정을 DB(`alarms` 테이블 등)에 저장하는 기능 및 관련 RLS 설계
- 여러 기기 간 알람/뉴스 설정 동기화(Supabase Realtime 활용)
- 뉴스 결과 캐싱(`news_cache` 테이블 등 DB 캐싱) — 비기능 요구사항에 따라 애초에 구현하지 않음
- 네이버 Clova Voice, Google Cloud TTS 등 유료/클라우드 TTS 연동 (1차 구현은 브라우저 내장 Web Speech API로 한정)
- 사용자별 관심 키워드 설정 UI 및 저장 기능

## 6. Open Questions

- **재생 타이밍 미확정**: 뉴스 음성 브리핑이 알람 소리와 정확히 어떤 시점/순서로 재생되는지 결정되지 않았다. (예: 알람 소리가 먼저 울리고 사용자가 끈 뒤 자동으로 뉴스가 재생되는지 / 알람 소리와 동시에 재생되는지 / 사용자가 버튼을 눌러야만 재생되는지) 이는 비기능 요구사항의 "알람 소리는 반드시 울려야 한다"는 원칙과 충돌하지 않도록 설계에 반영되어야 하므로, 구현 전 반드시 확정이 필요하다.
- **검색 키워드 미확정**: 명세서 예시 코드는 `query=속보`로 하드코딩되어 있으나, 이것이 실제 서비스에서 사용할 고정 키워드인지, 카테고리(정치/경제/IT 등) 기반으로 바꿀지, 향후 사용자가 직접 키워드를 지정할 수 있게 할지 결정되지 않았다.
- 뉴스 API 실패 시 화면에 어떤 대체 문구/상태를 보여줄지(예: "뉴스를 불러오지 못했습니다" 안내 문구 노출 여부 및 문구 내용) 구체적으로 정의되지 않았다.
- 뉴스 5건 중 일부만 조회에 성공한 경우(예: 3건만 반환) 이를 그대로 표시할지, 실패로 간주할지 정책이 없다.

## 7. 성공 기준

- 뉴스 조회/표시/TTS 재생 중 임의의 단계에서 강제로 오류를 발생시켜도, 알람 소리는 100% 정상적으로 재생된다.
- 네이버 API 키(Client ID/Secret)가 프론트엔드 번들, 브라우저 네트워크 탭, 클라이언트 로그 어디에도 노출되지 않는다(코드 리뷰 및 네트워크 검사로 확인).
- 정상 네트워크 환경에서 알람 화면 진입 시 최신 뉴스 5건이 노출되고, 매 진입마다 실시간으로 새로 조회된 결과가 반영된다(캐시된 이전 결과가 재사용되지 않음을 확인).
- 뉴스 데이터가 어떤 형태로도 DB에 영속 저장되지 않음을 확인한다(관련 테이블/저장 로직 부재).
- TTS를 통해 뉴스 제목이 한국어(`ko-KR`)로 정상 낭독된다.
